# Паспорт проекта CodeCrafters AI

## 1. Общие сведения о проекте

**Название проекта:** CodeCrafters AI.

**Тема ВКР:** «Разработка образовательного SaaS-сервиса с использованием ИИ для генерации заданий».

**Тип проекта:** образовательный SaaS-сервис, реализованный в виде веб-приложения с микросервисной backend-архитектурой.

**Назначение проекта:** разработка веб-сервиса, который помогает преподавателям детских онлайн-школ программирования быстрее создавать, редактировать, назначать и проверять учебные задания с использованием искусственного интеллекта.

**Краткое описание проекта:** CodeCrafters AI — это образовательный SaaS-сервис для преподавателей детских онлайн-школ программирования. Сервис позволяет преподавателю создавать учебные группы, принимать учеников по коду приглашения, генерировать задания с помощью GigaChat API, редактировать результат генерации, назначать задания группе и вручную проверять ответы учеников.

Проект ориентирован на создание минимально жизнеспособной версии продукта, достаточной для демонстрации основной бизнес-логики и защиты ВКР.

## 2. Актуальность проекта

Преподаватели детских онлайн-школ программирования регулярно сталкиваются с необходимостью подготовки учебных заданий, практических работ, тестов и домашних заданий для разных возрастных групп и уровней подготовки. Создание таких материалов вручную требует значительного времени, особенно если преподавателю необходимо адаптировать задания под конкретную тему, возраст учеников, направление обучения и уровень сложности.

Использование искусственного интеллекта позволяет автоматизировать часть методической работы преподавателя. При этом преподаватель сохраняет контроль над итоговым материалом: он может просмотреть, отредактировать и только после этого назначить задание ученикам. Такой подход снижает трудозатраты на подготовку материалов и повышает гибкость образовательного процесса.

SaaS-формат позволяет использовать сервис через браузер без установки дополнительного программного обеспечения. За счёт многопользовательской архитектуры несколько преподавателей могут работать в системе независимо друг от друга, управляя собственными группами, заданиями и учениками.

## 3. Целевая аудитория

Основной целевой аудиторией сервиса являются преподаватели детских онлайн-школ программирования.

Дополнительными пользователями являются ученики, которые получают доступ к назначенным заданиям, выполняют их и отправляют ответы преподавателю.

Потенциальные пользователи сервиса:

- преподаватели Scratch, Python, алгоритмики и HTML/CSS;
- наставники детских IT-школ;
- методисты, подготавливающие учебные материалы;
- ученики детских онлайн-школ программирования.

## 4. Учебные направления MVP

В рамках минимально жизнеспособной версии сервиса реализуется поддержка четырёх направлений обучения:

1. **Scratch** — визуальное программирование для младших школьников.
2. **Python** — текстовое программирование для школьников среднего и старшего возраста.
3. **Алгоритмика** — задания на логику, условия, циклы, переменные, блок-схемы и базовые алгоритмы.
4. **HTML/CSS** — создание простых веб-страниц и оформление интерфейсов.

Выбранный набор направлений позволяет охватить разные возрастные группы и разные форматы обучения программированию.

## 5. Типы генерируемых заданий

В MVP сервис поддерживает генерацию пяти типов учебных материалов:

1. **Практическое задание** — задание, в котором ученик должен выполнить конкретное действие: написать программу, создать сцену в Scratch, сверстать блок страницы или решить алгоритмическую задачу.
2. **Тест** — набор вопросов с вариантами ответов по выбранной теме.
3. **Задание на исправление ошибки** — задание, в котором ученик должен найти и исправить ошибку в коде, алгоритме или структуре решения.
4. **Творческий мини-проект** — более свободное задание, направленное на создание игры, сайта, викторины, анимации или интерактивной программы.
5. **Домашнее задание с критериями проверки** — учебное задание, включающее описание работы, ожидаемый результат и критерии оценивания для преподавателя.

Такой набор типов заданий позволяет использовать сервис как для коротких упражнений, так и для более комплексных практических работ.

## 6. Роли пользователей

В системе предусмотрены две основные роли пользователей:

### 6.1. Преподаватель

Преподаватель является основным пользователем сервиса. Он может:

- зарегистрироваться и войти в систему;
- создавать учебные группы;
- получать код приглашения для группы;
- генерировать задания с помощью ИИ;
- редактировать сгенерированные задания;
- назначать задания учебной группе;
- просматривать ответы учеников;
- устанавливать статус проверки;
- оставлять комментарии к работам учеников.

### 6.2. Ученик

Ученик является пользователем с ограниченным набором функций. Он может:

- зарегистрироваться и войти в систему;
- присоединиться к учебной группе по коду приглашения;
- просматривать назначенные задания;
- отправлять текстовый ответ и ссылку на выполненную работу;
- видеть статус проверки;
- повторно отправлять ответ, если преподаватель вернул работу на доработку.

### 6.3. Администратор

Отдельная роль администратора в MVP не реализуется. Возможность добавления административной панели рассматривается как направление дальнейшего развития проекта.

## 7. SaaS-модель проекта

CodeCrafters AI реализуется как многопользовательский образовательный SaaS-сервис, доступный через веб-интерфейс. Система поддерживает централизованное хранение данных, авторизацию пользователей и разграничение доступа на основе ролей.

Каждый преподаватель работает в собственном пространстве: создаёт свои учебные группы, генерирует свои задания, назначает их своим группам и проверяет ответы своих учеников. Данные разных преподавателей изолированы друг от друга: один преподаватель не имеет доступа к группам, заданиям и ученикам другого преподавателя.

Оплата, тарифные планы и биллинг в MVP не реализуются. Эти функции могут быть добавлены на следующих этапах развития сервиса.

## 8. Основной пользовательский сценарий преподавателя

Основной сценарий работы преподавателя выглядит следующим образом:

1. Преподаватель регистрируется или входит в систему.
2. Создаёт учебную группу.
3. Получает код приглашения и передаёт его ученикам.
4. Переходит в раздел генерации задания.
5. Указывает параметры генерации: направление, тему, возрастную категорию, уровень сложности, тип задания и дополнительные пожелания.
6. Система отправляет запрос в AI Generation Service.
7. AI Generation Service обращается к GigaChat API и получает структурированный результат.
8. Преподаватель просматривает сгенерированное задание.
9. При необходимости редактирует поля задания.
10. Назначает задание учебной группе и при желании указывает дедлайн.
11. Ученики группы получают доступ к заданию.
12. Преподаватель просматривает отправленные ответы, ставит статус проверки и оставляет комментарий.

## 9. Основной пользовательский сценарий ученика

Основной сценарий работы ученика выглядит следующим образом:

1. Ученик регистрируется или входит в систему.
2. Вводит код приглашения, полученный от преподавателя.
3. Присоединяется к учебной группе.
4. Открывает список назначенных заданий.
5. Выбирает задание и изучает его описание.
6. Выполняет работу во внешней среде или локально.
7. Отправляет текстовый ответ и при необходимости ссылку на результат.
8. Ожидает проверки преподавателем.
9. Видит статус работы: «Отправлено», «Зачтено» или «Нужно доработать».
10. Если работа возвращена на доработку, исправляет её и отправляет новый вариант ответа.

## 10. Структура результата генерации задания

Результатом работы модуля ИИ-генерации является структурированное учебное задание, включающее следующие поля:

- название задания;
- описание для ученика;
- цель задания;
- уровень сложности;
- инструкция выполнения;
- ожидаемый результат;
- критерии проверки;
- подсказки для ученика;
- пример решения или эталонный ответ для преподавателя.

Часть информации предназначена для ученика, а часть — только для преподавателя. Например, пример решения и подробные критерии проверки могут отображаться только преподавателю.

## 11. Параметры генерации задания

Для генерации задания преподаватель заполняет форму со следующими параметрами:

- направление обучения: Scratch, Python, алгоритмика, HTML/CSS;
- тема задания: свободное текстовое поле;
- возрастная категория: 7–9, 10–12, 13–15, 16–17 лет;
- уровень сложности: начальный, средний, продвинутый;
- тип задания: практическое задание, тест, задание на исправление ошибки, творческий мини-проект, домашнее задание с критериями проверки;
- количество вопросов: используется только для типа «Тест»;
- дополнительные пожелания преподавателя.

На основе этих параметров система формирует запрос к GigaChat API и получает структурированный учебный материал.

## 12. Уровни сложности

В сервисе используются три уровня сложности:

1. **Начальный** — для учеников, которые только знакомятся с темой.
2. **Средний** — для учеников, которые уже знают основы и могут выполнять задания из нескольких шагов.
3. **Продвинутый** — для учеников, которым можно давать более самостоятельные и творческие задачи.

Выбор уровня сложности влияет на формулировку задания, количество шагов, объём подсказок и требования к результату.

## 13. Возрастные категории

В сервисе используются четыре возрастные категории:

1. **7–9 лет** — младшие школьники, простая алгоритмика и Scratch.
2. **10–12 лет** — Scratch, алгоритмика, первые задания по Python и HTML/CSS.
3. **13–15 лет** — Python, HTML/CSS, алгоритмические задачи и мини-проекты.
4. **16–17 лет** — более сложные практические задания, самостоятельные мини-проекты и задачи на закрепление навыков разработки.

Выбор возрастной категории влияет на сложность формулировок, объём задания, уровень самостоятельности ученика и требования к результату.

## 14. Регистрация и авторизация

В системе используется единая форма регистрации с выбором роли пользователя. При регистрации пользователь указывает:

- имя;
- email;
- пароль;
- роль: преподаватель или ученик.

После регистрации пользователь получает доступ к интерфейсу, соответствующему его роли.

В MVP используется авторизация на основе JWT access token. После успешного входа пользователь получает токен, который frontend-приложение передаёт в защищённые запросы к backend-сервисам.

JWT должен содержать:

- идентификатор пользователя;
- email;
- роль пользователя.

Refresh-токены в MVP не реализуются. Их добавление рассматривается как перспектива развития проекта.

## 15. Работа с учебными группами

Преподаватель может создавать учебные группы. При создании группы указываются:

- название группы;
- описание группы.

После создания группы система автоматически формирует уникальный код приглашения. Преподаватель передаёт этот код ученикам. Ученик вводит код в личном кабинете и присоединяется к соответствующей группе.

Механизм присоединения по коду позволяет отказаться от сложной email-рассылки приглашений и при этом сохранить понятную модель подключения учеников к группам.

## 16. Назначение заданий группе

В MVP задания назначаются только учебной группе. Индивидуальные назначения отдельным ученикам не реализуются.

После генерации и редактирования задания преподаватель выбирает группу, которой необходимо выдать задание. При назначении преподаватель может указать дедлайн. Дедлайн является необязательным параметром.

Если дедлайн указан, он отображается ученикам в личном кабинете. Если дедлайн не указан, задание считается бессрочным.

## 17. Отправка и проверка ответов

Ученик отправляет результат выполнения задания в виде:

- текстового ответа;
- необязательной ссылки на внешний ресурс.

Ссылка может вести на Scratch-проект, Replit, CodePen, GitHub Pages, Google Docs или другой внешний ресурс.

Проверка ответов в MVP выполняется вручную преподавателем. Преподаватель просматривает текстовый ответ и ссылку, затем устанавливает статус проверки и при необходимости оставляет комментарий.

## 18. Статусы выполнения задания

В системе используются следующие статусы выполнения задания:

1. **Не начато** — ученик ещё не отправлял ответ.
2. **Отправлено** — ученик отправил ответ, работа ожидает проверки.
3. **Зачтено** — преподаватель проверил работу и принял её.
4. **Нужно доработать** — преподаватель оставил комментарий и вернул работу на исправление.

Если работа возвращена на доработку, ученик может повторно отправить ответ. В MVP хранится только последняя версия ответа без истории всех попыток.

## 19. Интерфейс пользователя

Интерфейс CodeCrafters AI выполняется в современном светлом стиле. Для навигации используется боковое меню, а основные данные отображаются в виде карточек, таблиц и форм.

### 19.1. Экраны преподавателя

Для преподавателя реализуются следующие экраны:

1. Страница входа и регистрации.
2. Панель преподавателя.
3. Страница групп.
4. Страница создания и редактирования группы.
5. Страница генерации задания.
6. Страница просмотра и редактирования сгенерированного задания.
7. Страница назначенных заданий и ответов учеников.

Отдельная библиотека заданий в MVP не реализуется. Сгенерированные задания сохраняются в системе в рамках назначений преподавателя. Полноценная библиотека заданий может быть добавлена в дальнейшем развитии проекта.

### 19.2. Экраны ученика

Для ученика реализуются следующие экраны:

1. Страница входа.
2. Панель ученика.
3. Страница присоединения к группе по коду.
4. Список назначенных заданий.
5. Страница выполнения задания.
6. История отправленных ответов.

Отдельная страница профиля пользователя в MVP не реализуется. В системе хранятся только базовые данные пользователя: имя, email, пароль и роль.

## 20. Архитектура приложения

Проект реализуется как микросервисное приложение.

### 20.1. Frontend

Frontend-приложение разрабатывается на React и TypeScript. Оно предоставляет интерфейс для преподавателя и ученика и взаимодействует с backend-сервисами через REST API.

### 20.2. Backend

Backend состоит из трёх микросервисов на Java Spring Boot:

1. **Auth Service** — регистрация, вход, JWT-авторизация и управление ролями пользователей.
2. **Assignment Service** — группы, участники групп, задания, назначение заданий, ответы учеников и ручная проверка.
3. **AI Generation Service** — формирование промпта, обращение к GigaChat API, обработка ответа ИИ и возврат структурированного задания.

### 20.3. База данных

В качестве базы данных используется PostgreSQL. В MVP применяется единая база данных для всех backend-сервисов, но таблицы логически разделяются по зонам ответственности сервисов.

Такой подход упрощает разработку, тестирование и развёртывание приложения. В дальнейшем проект может быть переведён на архитектуру database per service, при которой каждый микросервис использует собственное хранилище данных.

### 20.4. Взаимодействие сервисов

Взаимодействие между backend-сервисами реализуется через REST API.

Основной сценарий межсервисного взаимодействия:

1. Frontend отправляет параметры генерации в Assignment Service.
2. Assignment Service отправляет REST-запрос в AI Generation Service.
3. AI Generation Service обращается к GigaChat API.
4. GigaChat API возвращает результат генерации.
5. AI Generation Service преобразует ответ в структуру задания и возвращает его Assignment Service.
6. Assignment Service сохраняет задание в PostgreSQL и возвращает результат frontend-приложению.

## 22. Интеграция с GigaChat API

В качестве внешнего сервиса искусственного интеллекта используется GigaChat API.

AI Generation Service принимает параметры задания от Assignment Service, формирует промпт и отправляет запрос к GigaChat API. В ответ сервис получает структурированное учебное задание, которое затем передаётся в Assignment Service.

Секретные данные, необходимые для обращения к GigaChat API, хранятся в переменных окружения и не включаются в исходный код проекта.

Пример переменных окружения:

- `GIGACHAT_AUTH_KEY`;
- `GIGACHAT_SCOPE`;
- `GIGACHAT_OAUTH_URL`;
- `GIGACHAT_API_URL`;
- `GIGACHAT_MODEL`;
- `AI_MOCK_ENABLED`.

## 23. Dev fallback-режим генерации

В AI Generation Service предусматривается dev fallback-режим. Он включается через переменную окружения:

`AI_MOCK_ENABLED=true`

При включённом fallback-режиме сервис возвращает демонстрационное структурированное задание без обращения к GigaChat API. Этот режим используется только для локальной разработки, тестирования и демонстрации системы при отсутствии доступа к внешнему API.

Основной режим работы сервиса — реальная интеграция с GigaChat API.

## 24. Развёртывание проекта

Для запуска и демонстрации системы используется Docker Compose.

В составе `docker-compose.yml` должны быть следующие контейнеры:

1. `postgres` — база данных PostgreSQL.
2. `auth-service` — сервис авторизации.
3. `assignment-service` — сервис заданий и групп.
4. `ai-generation-service` — сервис генерации заданий через ИИ.
5. `frontend` — React-приложение.

Проект должен запускаться одной командой:

```bash
docker compose up --build
```

Такой подход обеспечивает воспроизводимую среду запуска и упрощает демонстрацию проекта.

## 25. Миграции базы данных

Для управления структурой базы данных используется Flyway.

Миграции распределяются между backend-сервисами по зонам ответственности:

- Auth Service создаёт таблицы, связанные с пользователями и авторизацией;
- Assignment Service создаёт таблицы групп, заданий, назначений и ответов;
- AI Generation Service создаёт таблицы, связанные с запросами генерации и логами обращений к ИИ, если такие данные сохраняются.

Каждый сервис создаёт только свои таблицы и не должен изменять таблицы другого сервиса.

## 27. Ограничения MVP

В MVP не реализуются следующие функции:

- отдельная административная панель;
- тарифы, оплата и биллинг;
- полноценная библиотека заданий;
- индивидуальное назначение заданий отдельным ученикам;
- загрузка файлов;
- автоматическая проверка решений с помощью ИИ;
- история всех попыток отправки ответа;
- refresh-токены;
- отдельная страница профиля пользователя;
- email-рассылки и приглашения по email.

Эти ограничения выбраны для снижения сложности реализации и концентрации на основной теме проекта: генерации учебных заданий с использованием ИИ в образовательном SaaS-сервисе.

Проект CodeCrafters AI соответствует теме ВКР, так как включает:
